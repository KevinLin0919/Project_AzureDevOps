# Implementation Plan: Asgards — Azure DevOps Python Package

**Branch**: `002-asgards-azure-devops-package` | **Date**: 2026-04-15 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/002-asgards-azure-devops-package/spec.md`

## Summary

Build `asgards`, a pip-installable Python package that wraps the Azure DevOps REST API
into six domain-specific client classes (Project, Repo, Member, Pipeline, Release, Branch).
The implementation uses the official `azure-devops` Python SDK as the primary interface,
falling back to `requests` only for endpoints not covered by the SDK (branch policy creation,
retention settings). All clients authenticate via PAT passed through constructor or env var.

## Technical Context

**Language/Version**: Python 3.11
**Primary Dependencies**: `azure-devops>=7.1.0b4`, `msrest>=0.7.1`, `requests>=2.31`
**Storage**: N/A — pure API wrapper, no local persistence
**Testing**: `pytest` + `pytest-cov`, with `unittest.mock` for mocking Azure SDK clients
**Target Platform**: Any Python 3.11 environment; CI runs on `ubuntu-latest`
**Project Type**: library (pip-installable Python package)
**Performance Goals**: N/A — latency is Azure DevOps API-bound; no local compute bottlenecks
**Constraints**: 5 MB upload hard limit (enforced client-side before API call); PAT-only auth
**Scale/Scope**: 6 modules, ~6 classes, ~35 public methods total

`tests/` at the same level. `setup.py` at `asgards/` root so `pip install git+...` resolves
the package correctly. This matches both the reference material and Constitution Principle IV.
